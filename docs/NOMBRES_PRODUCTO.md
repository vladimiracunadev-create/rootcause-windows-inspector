# Nombres posibles del producto

Este documento propone nombres para el producto y explica por qué conviene evitar algunos nombres demasiado genéricos o demasiado parecidos a utilidades ya existentes.

## Criterios usados

El nombre ideal para este software debería comunicar varias cosas a la vez:

- que trabaja sobre **Windows**,
- que encuentra la **causa real** del problema,
- que permite una acción **rápida o de un clic**,
- que no suena a “limpiador mágico” sino a herramienta seria,
- que puede crecer a una línea de producto más profesional.

## Nombres que conviene evitar

### RootCause
Aunque comunica vigilancia de señales del sistema, ya aparece usado por otros productos y proyectos en la web, por lo que no es una opción ideal como marca principal.

### WinPulse
También aparece usado históricamente por otras utilidades y proyectos, por lo que conviene evitarlo como nombre principal.

### OneClick
Es demasiado genérico y ya se usa en muchos contextos de software y despliegue, por lo que sería débil como marca y difícil de defender.

## Recomendación de enfoque

Conviene separar:

- **nombre de marca**,
- **descriptor funcional**.

Ejemplo:

- **Marca**: `RootCause`
- **Descriptor**: `Windows Performance Inspector`

Así el producto puede sonar profesional y, al mismo tiempo, explicar de inmediato qué hace.

## Alternativas propuestas

### 1. RootCause
**Fortaleza:** suena serio, orientado a causa raíz y compatible con el mensaje “resolver con un clic”.

**Descriptor sugerido:** `RootCause – Windows Slowdown Inspector`

### 2. WinCause
**Fortaleza:** corto, muy directo, recuerda Windows + causa.

**Riesgo:** en algunos mercados el término aparece asociado a otros usos de software, así que requiere revisión legal más profunda antes de adoptarlo.

### 3. CauseClick
**Fortaleza:** comunica causa + acción simple.

**Descriptor sugerido:** `CauseClick – One-Click Windows Diagnosis`

### 4. RootClick
**Fortaleza:** suena moderno y directo, mezcla causa raíz con intervención rápida.

**Descriptor sugerido:** `RootClick – Windows Conflict Finder`

### 5. TempTrace
**Fortaleza:** muy bueno si quieres destacar el problema real de temporales, cachés y rastreo de actividad.

**Límite:** se queda corto si luego el producto crece a red, memoria, ETW y servicios.

### 6. DiskCause
**Fortaleza:** excelente si el foco comercial inicial será “disco al 100% / archivos temporales / lentitud”.

**Límite:** podría sonar demasiado centrado en disco y no abarcar memoria/red.

### 7. ConflictLens
**Fortaleza:** comunica visibilidad y análisis. Sirve bien para una estética moderna.

**Descriptor sugerido:** `ConflictLens – Windows Performance Root Cause`

### 8. CauseRadar
**Fortaleza:** transmite detección continua y rastreo de problemas.

**Descriptor sugerido:** `CauseRadar – Windows Slow PC Detector`

### 9. RootSignal
**Fortaleza:** mantiene parte del espíritu del nombre actual, pero orientado más a causa raíz.

**Límite:** todavía puede sonar cercano a nombres de monitoreo ya usados, así que requiere revisión adicional.

### 10. FixScope
**Fortaleza:** corto, profesional, escalable y útil para línea de producto.

**Descriptor sugerido:** `FixScope – Windows Issue Cause Analyzer`

## Mis favoritas

Si el objetivo real es **un producto profesional, entendible y vendible**, mis opciones más fuertes son:

1. **RootCause**
2. **ConflictLens**
3. **FixScope**
4. **CauseRadar**
5. **RootClick**

## Recomendación práctica

### Si quieres sonar técnico y serio
- **RootCause**
- **ConflictLens**
- **FixScope**

### Si quieres sonar más comercial y fácil de recordar
- **CauseRadar**
- **RootClick**
- **CauseClick**

### Si quieres sonar muy enfocado al problema de lentitud por temporales/disco
- **DiskCause**
- **TempTrace**

## Nombre recomendado para este repositorio, por ahora

Mientras no se haga una revisión de marca más formal, la mejor ruta es usar un nombre de trabajo como:

> **RootCause**
>
> **RootCause – Windows Performance Inspector**

## Nota importante

La revisión realizada aquí es **preliminar y orientativa**. Antes de publicar el producto como marca definitiva, conviene revisar:

- disponibilidad de dominio,
- disponibilidad de usuario en GitHub/GitLab,
- coincidencias comerciales claras en buscadores,
- y, si el proyecto crece comercialmente, búsqueda formal de marca en los países objetivo.
